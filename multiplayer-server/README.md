# CyberWorld Multiplayer Server

Backend service for **Operation Starshield** — multiplayer progression, email auth, chapter/subscription gating, gameplay state, and Stripe billing.

## Features

- Email/password registration and login (bcrypt + JWT)
- Chapter gate policy:
  - Chapters 1–3: open to everyone
  - Chapter 4: requires authenticated account (free tier)
  - Chapter 5+: requires basic subscription
- Mission reward system: XP, tool unlocks, companion discovery, faction reputation
- Stripe Checkout + webhook billing (falls back to mock mode when keys are absent)
- Dual persistence: PostgreSQL (production) or db.json (dev)
- Socket.IO realtime presence, lobby, and chat

## Quick Start (Local Dev)

```bash
cd multiplayer-server
cp .env.example .env        # edit JWT_SECRET at minimum
npm install
npm run dev                  # starts on http://localhost:8787
```

## Environment Variables

| Variable | Required | Description |
|---|---|---|
| `PORT` | No | Server port (default `8787`) |
| `JWT_SECRET` | **Yes** | Token signing secret — `openssl rand -base64 32` to generate |
| `APP_ORIGIN` | **Yes** | Comma-separated allowed CORS origins |
| `DATABASE_URL` | No | PostgreSQL connection string — omit for db.json fallback |
| `STRIPE_SECRET_KEY` | No | Stripe secret key — omit for mock billing |
| `STRIPE_WEBHOOK_SECRET` | No | Stripe CLI or dashboard webhook signing secret |
| `STRIPE_PRICE_ID` | No | Stripe Price ID for the basic subscription product |

## Deploy to Render

1. Push to GitHub — the `render.yaml` in this directory defines the service.
2. In Render Dashboard → **New** → **Blueprint** → select this repo.
3. Render auto-creates the `cyberworld-multiplayer` web service.
4. In the Render service **Environment** tab, set:
   - `DATABASE_URL` — add a Render PostgreSQL database, copy its Internal URL here
   - `STRIPE_SECRET_KEY` / `STRIPE_WEBHOOK_SECRET` / `STRIPE_PRICE_ID` — from your Stripe Dashboard
   - `JWT_SECRET` is auto-generated by `render.yaml`
   - `APP_ORIGIN` is preset to `https://personfu.github.io,https://fllc.net`
5. Deploy. The service URL will be `https://cyberworld-multiplayer.onrender.com`.

### Stripe Webhook Setup

1. In Stripe Dashboard → Developers → Webhooks → **Add endpoint**
2. URL: `https://cyberworld-multiplayer.onrender.com/api/stripe/webhook`
3. Events: `checkout.session.completed`, `customer.subscription.deleted`
4. Copy the signing secret → set as `STRIPE_WEBHOOK_SECRET` on Render

## API Reference

### Auth

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/auth/register` | — | `{email, password, displayName}` → `{token, user}` |
| `POST` | `/api/auth/login` | — | `{email, password}` → `{token, user}` |
| `GET` | `/api/auth/me` | Bearer | Returns authenticated user |

### World & Gates

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/api/world/state` | Optional | World status + current user if logged in |
| `GET` | `/api/gate?chapter=N` | Optional | Check if user can access chapter N |

### Progress & Gameplay

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/api/progress` | Bearer | User progress, inventory, tier |
| `POST` | `/api/progress` | Bearer | `{chapter, missionId, xpDelta}` — applies mission rewards |

Mission rewards (XP, tools, companions, faction rep) are auto-applied when `missionId` is first cleared.

### Subscription

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/subscription/create-checkout` | Bearer | Creates Stripe Checkout session (or mock URL) |
| `POST` | `/api/subscription/activate-basic` | Bearer | Mock activation (disabled when Stripe is configured) |
| `POST` | `/api/stripe/webhook` | Stripe sig | Processes `checkout.session.completed` and `customer.subscription.deleted` |

### System

| Method | Path | Description |
|---|---|---|
| `GET` | `/health` | `{ok: true, service, timestamp}` |

## Smoke Test

```bash
# Against local dev
node smoke-test.js

# Against production
node smoke-test.js https://cyberworld-multiplayer.onrender.com
```

The smoke test verifies: health, register, login, me, gate checks (ch3/4/5), mission rewards + inventory + faction rep, subscription flow, and world state.

## Rollback

1. In Render Dashboard → service → **Deploys** → select a previous deploy → **Rollback**
2. Database schema is additive — rolling back code won't break existing data
3. To fully reset a Render Postgres: connect via `psql` and `DROP TABLE users, world_state;`
